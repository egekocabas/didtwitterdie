## Problem

The Google Trends section is currently a "Coming soon" placeholder. Google Trends data shows search interest — a different signal from Tranco's domain ranking. DNS/domain data measures what people type in the URL bar; Google Trends measures what people search for on Google.

## Research Summary

### There is no official Google Trends API

All programmatic approaches use unofficial widget endpoints that:
- Require session cookies (bare requests get 429 instantly)
- Aggressively rate-limit even with cookies (429 after 2-3 rapid calls)
- May block Cloudflare Worker shared IPs
- Can break without notice if Google changes the endpoints

### Live API test results (verified 2026-03-28)

| Keywords | twitter | x.com | x twitter | x social media |
|----------|---------|-------|-----------|----------------|
| 2022 baseline | ~76-80 | 0 | 0 | 0 |
| Rebrand week (2023-07-23) | 84 | 1 | 3 | 0 |
| Current (2026-03) | 31-32 | 1 | 2 | 0 |

**Key finding**: ALL X-related search terms have near-zero interest (0-3). Nobody Googles "x.com" or "x social media" — they type it directly. "twitter" is still overwhelmingly dominant in search, though declining from ~80 to ~32.

### Recommended approach: Google Trends Embed Widget

Instead of scraping the API (fragile, rate-limited, cookie-dependent), **embed Google's official widget** directly:

**Advantages:**
- No backend changes — no `fetchTrendsData()`, no API calls, no rate limits
- Always up-to-date — Google serves fresh data on each page load
- Officially supported — designed for third-party embedding
- No API keys, no cookies, no scraping
- Interactive — users can hover for exact values (Google provides this built-in)

**Trade-offs:**
- Can't style with our Recharts/dark mode — uses Google's own light-themed UI
- Can't use trends data in verdict calculation (but we don't need to — verdict uses Tranco + Radar)
- Slightly different visual style from our other charts

**Why the trade-offs are acceptable:**
- The section already looks "different" as a third-party data source — Google branding adds credibility
- We wrap it in our `ChartWrapper` for consistent section framing (title, description, scroll animation)
- Dark mode: the white iframe stands out, but we can add a subtle border/background to make it look intentional

### Keywords for the embed

Show **"twitter" vs "x.com"** — even though x.com is near-zero, it tells the story: "people still overwhelmingly search for 'twitter' and nobody Googles 'x.com'."

## Steps

### Step 1: Rewrite `src/components/TrendsChart.jsx`

Replace the dashed-border placeholder with a Google Trends embed iframe.

```jsx
import ChartWrapper from "./ChartWrapper";

const TRENDS_EMBED_URL =
  "https://trends.google.com/trends/embed/explore/TIMESERIES?" +
  "req=" + encodeURIComponent(JSON.stringify({
    comparisonItem: [
      { keyword: "twitter", geo: "", time: "2022-01-01 2026-12-31" },
      { keyword: "x.com", geo: "", time: "2022-01-01 2026-12-31" },
    ],
    category: 0,
    property: "",
  })) +
  "&tz=0";

export default function TrendsChart() {
  return (
    <ChartWrapper
      title="What people Google"
      description='Google search interest for "twitter" vs "x.com" over time. Values are relative (0-100), not absolute search volume.'
    >
      <div className="rounded-xl overflow-hidden border border-gray-200 dark:border-gray-700 bg-white">
        <iframe
          src={TRENDS_EMBED_URL}
          width="100%"
          height="400"
          frameBorder="0"
          loading="lazy"
          title="Google Trends: twitter vs x.com"
        />
      </div>
      <p className="text-xs text-gray-400 dark:text-gray-500 mt-2 text-center">
        Chart powered by Google Trends
      </p>
    </ChartWrapper>
  );
}
```

**Key details:**
- `loading="lazy"` — don't load the iframe until it scrolls into view
- White background on the wrapper to match Google's embed theme
- Rounded corners + border to frame the embed cleanly
- Small "Chart powered by Google Trends" credit below
- No props needed — the embed is self-contained
- `TrendsChart` keeps zero props (no changes to App.jsx needed!)
